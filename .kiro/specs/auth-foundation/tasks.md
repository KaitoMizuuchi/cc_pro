# Implementation Plan

- [ ] 1. バックエンド基盤セットアップ
- [x] 1.1 PrismaClient シングルトンを作成する
  - バックエンド全体で共有する PrismaClient インスタンスをシングルトンパターンで生成する
  - 開発時のホットリロードで複数インスタンスが生成されないよう、グローバル変数にキャッシュする仕組みを導入する
  - _Requirements: 1.12_

- [x] 1.2 (P) JWT_SECRET 環境変数の設定と起動時チェックを追加する
  - バックエンドの環境変数に JWT_SECRET を追加し、未設定時はサーバー起動を中断してエラーメッセージを出力する
  - .env.example にキー名を追記し、開発者が必要な環境変数を把握できるようにする
  - _Requirements: 2.2, 2.6_

- [ ] 2. バリデーションスキーマを定義する
- [ ] 2.1 (P) バックエンド用バリデーションスキーマを作成する
  - サインアップリクエスト用スキーマを定義する（name: 必須、email: 必須かつメール形式、password: 必須かつ8文字以上かつ半角英字・数字・記号を各1文字以上）
  - ログインリクエスト用スキーマを定義する（email: 必須、password: 必須）
  - バリデーションメッセージは日本語で記述する
  - _Requirements: 1.3, 1.4, 1.5, 1.7, 2.3, 2.4_

- [ ] 2.2 (P) フロントエンド用バリデーションスキーマを作成する
  - サインアップフォーム用スキーマを定義する（name、email、password、passwordConfirm の4フィールド）
  - パスワード確認フィールドの一致検証を refine で実装する
  - ログインフォーム用スキーマを定義する（email、password の2フィールド）
  - バリデーションメッセージは日本語で記述する
  - _Requirements: 1.3, 1.4, 1.5, 1.6, 1.7, 2.3, 2.4_

- [ ] 3. バックエンド認証サービスを実装する
- [ ] 3.1 サインアップ機能を実装する
  - メールアドレスの重複チェックを行い、既に登録済みの場合はエラーコード EMAIL_ALREADY_EXISTS を返却する
  - パスワードを Bun.password で bcrypt ハッシュ化し、新しいユーザーレコードをデータベースに作成する
  - 成功時はユーザーの ID・名前・メールアドレスを返却する（パスワードは返却しない）
  - タスク1.1 で作成した PrismaClient シングルトンを使用する
  - _Requirements: 1.2, 1.8, 1.12_

- [ ] 3.2 ログイン機能を実装する
  - メールアドレスでユーザーを検索し、存在しない場合は統一エラーメッセージ（INVALID_CREDENTIALS）を返却する
  - Bun.password.verify でパスワードを検証し、不一致の場合も同一のエラーメッセージを返却する（どちらが間違いかを明かさない）
  - 認証成功時に hono/jwt の sign で JWT トークンを生成する（ペイロード: sub, email, exp=24時間）
  - トークンとユーザー情報を返却する
  - _Requirements: 2.2, 2.5_

- [ ] 4. バックエンド認証 API ルートを実装する
- [ ] 4.1 サインアップエンドポイントを作成する（POST /api/auth/signup）
  - リクエストボディをバリデーションスキーマで検証し、失敗時は 400 レスポンスを返却する
  - 認証サービスのサインアップ機能を呼び出し、結果に応じたステータスコード（201 / 409 / 500）を返却する
  - エラーレスポンスは統一フォーマット（code + message）で返却する
  - _Requirements: 1.2, 1.8, 1.10, 5.1_

- [ ] 4.2 ログインエンドポイントを作成する（POST /api/auth/login）
  - リクエストボディをバリデーションスキーマで検証し、失敗時は 400 レスポンスを返却する
  - 認証サービスのログイン機能を呼び出し、成功時は JWT トークンを httpOnly Cookie（auth_token）にセットする
  - Cookie の属性を設定する（HttpOnly, SameSite=Lax, Path=/, Max-Age=86400）
  - レスポンスボディにはユーザー情報のみ含め、トークンは含めない
  - _Requirements: 2.2, 2.5, 2.6, 5.2_

- [ ] 4.3 ログアウトエンドポイントを作成する（POST /api/auth/logout）
  - auth_token Cookie をクリアする（Max-Age=0）
  - 成功メッセージを返却する
  - _Requirements: 3.2_

- [ ] 4.4 認証状態確認エンドポイントを作成する（GET /api/auth/me）
  - JWT ペイロードからユーザー ID を取得し、データベースからユーザー情報を検索する
  - ユーザーの ID・名前・メールアドレスを返却する
  - このエンドポイントは認証ミドルウェアの保護下に置く
  - _Requirements: 4.5_

- [ ] 4.5 認証ルートをメインサーバーにマウントする
  - 作成した認証ルートを Hono アプリケーションの /api/auth パスにマウントする
  - _Requirements: 2.2_

- [ ] 5. バックエンド認証ミドルウェアを実装する
  - リクエストの Cookie から auth_token を取得し、hono/jwt の verify でトークンを検証する
  - トークンが存在しない場合は TOKEN_MISSING エラー（401）を返却する
  - トークンが無効または期限切れの場合は TOKEN_INVALID / TOKEN_EXPIRED エラー（401）を返却する
  - 検証成功時はデコード済みペイロードをリクエストコンテキストに設定し、後続のハンドラから参照できるようにする
  - 認証不要パス（/api/auth/signup, /api/auth/login）を除外する
  - ミドルウェアを /api パス配下の全ルートに適用する
  - _Requirements: 3.3, 4.2, 4.3_

- [ ] 6. フロントエンド基盤セットアップ
- [ ] 6.1 トースト通知ライブラリを導入する
  - sonner パッケージをインストールし、アプリケーションのルートに Toaster コンポーネントを配置する
  - 成功・エラー通知が正しく表示されることを確認する
  - _Requirements: 1.9, 1.10, 5.1, 5.2_

- [ ] 6.2 API クライアントを認証対応に拡張する
  - 既存の API クライアントに credentials: "include" を追加し、Cookie を自動送信するようにする
  - 401 レスポンスを受信した場合にログインページへリダイレクトする仕組みを追加する
  - 既存の API クライアントインターフェースとの後方互換性を維持する
  - _Requirements: 4.4_

- [ ] 7. フロントエンド認証サービス層と状態管理を実装する
- [ ] 7.1 認証サービス層を作成する
  - サインアップ・ログイン・ログアウト・認証状態確認の各 API 呼び出し関数を定義する
  - API クライアントを使用してリクエストを送信する
  - _Requirements: 2.2, 2.6, 3.2_

- [ ] 7.2 認証コンテキストとフックを作成する
  - React Context で認証状態（ユーザー情報・認証済みフラグ・ローディング状態）を管理する AuthProvider を作成する
  - 初期化時に認証状態確認 API を呼び出し、Cookie ベースでログイン状態を復元する
  - ログイン成功時にユーザー情報をコンテキストに保存し、ログアウト時にクリアする機能を提供する
  - useAuth フックで AuthProvider の値にアクセスできるようにする
  - _Requirements: 2.2, 3.2, 4.4, 4.5_

- [ ] 8. フロントエンド認証ページを実装する
- [ ] 8.1 (P) サインアップページとフォームを作成する
  - ユーザー名・メールアドレス・パスワード・パスワード確認の入力フォームを表示する
  - react-hook-form と zod リゾルバを連携し、フィールドレベルのバリデーションエラーを表示する
  - 送信時に認証サービスのサインアップ関数を TanStack Query の useMutation 経由で呼び出す
  - 成功時にトースト通知（「アカウントが作成されました。ログインしてください」）を表示し、ログインページに遷移する
  - 失敗時にエラー内容に応じたトースト通知を表示する（メール重複: 「このメールアドレスは既に登録されています」、サーバーエラー: 「アカウントの作成に失敗しました。再度お試しください」）
  - 送信中はボタンを無効化し、ローディング状態を表示する
  - ログインページへの遷移リンク（「ログインに戻る」）を配置する
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 1.6, 1.7, 1.8, 1.9, 1.10, 1.11, 5.1, 5.4_

- [ ] 8.2 (P) ログインページとフォームを作成する
  - メールアドレス・パスワードの入力フォームを表示する
  - react-hook-form と zod リゾルバを連携し、フィールドレベルのバリデーションエラーを表示する
  - 送信時に認証サービスのログイン関数を TanStack Query の useMutation 経由で呼び出す
  - 成功時に useAuth の login でユーザー情報をコンテキストに保存し、管理ページに遷移する
  - 失敗時にエラー内容に応じたトースト通知を表示する（認証失敗: 「メールアドレスまたはパスワードが正しくありません」、サーバーエラー: 「ログインに失敗しました。再度お試しください」）
  - 送信中はボタンを無効化し、ローディング状態を表示する
  - サインアップページへの遷移リンクを配置する
  - _Requirements: 2.1, 2.2, 2.3, 2.4, 2.5, 2.7, 5.2, 5.4_

- [ ] 9.2 (P) ゲストガードを作成する

- [ ] 9.3 (P) サイドナビゲーションを作成する

  - ログアウトボタンを配置し、クリック時に認証サービスのログアウト関数を呼び出す
  - ログアウト成功時にコンテキストをクリアし、ログインページに遷移する
  - ログアウト処理中にエラーが発生した場合でも、クライアント側でコンテキストをクリアしログインページに遷移するフォールバック処理を実装する

- [ ] 10. ルーティング統合とトークン有効期限ハンドリング
- [ ] 10.1 アプリケーション全体のルーティングを構成する
  - AuthProvider をアプリケーションのルート近くに配置する

  - _Requirements: 4.4, 3.3_
